Use Case

Use Case — Employee Support Agent

A realistic enterprise scenario that works naturally across the full Microsoft Foundry journey.

Story

You work in a company. Employees ask questions every day such as:

  • How do I request vacation?
  • How do I reset my password?
  • Where is the travel policy?
  • Who approves expenses?

The AI Agent should

  • Answer questions
  • Search documents
  • Create IT tickets using a mock API
  • Remember employee preferences
  • Follow company policies

Challenge Details

Detailed Challenge Structure

Each challenge follows the same workshop pattern: time, objectives, context, build approach, get started steps, and success criteria.

🤖

Challenge 1: Build the Agent and Choose the Model

Time: ~30 minutes

Objectives: Build the first Employee Support Agent, compare a stronger GPT model with a smaller model, and choose the best fit for enterprise support.

Context: Start with a basic assistant that answers common employee questions and establish the model quality, latency, and cost tradeoffs.

Portal or SDK? Use the SDK for repeatable setup and then inspect the created agent in the Foundry portal.

Get Started: Create the agent, test employee support prompts, compare outputs from multiple models, and document your chosen baseline.

Success Criteria: A working baseline agent exists, model comparisons are complete, and a justified model choice is recorded.

📚

Challenge 2: Grounding

Time: ~30 minutes

Objectives: Attach HR and IT documents so the agent can answer based on company knowledge instead of general model memory.

Context: Employees ask policy and process questions that must be answered accurately from trusted internal sources.

Portal or SDK? Use Foundry knowledge and retrieval features through the SDK or portal workflow.

Agents and Tools: Introduce file search, vector search, and grounded answer generation.

Get Started: Upload synthetic HR and IT documents, create a knowledge source, connect it to the agent, and test retrieval-based questions.

Success Criteria: Answers cite relevant internal policy content and improve over the ungrounded baseline.

🛠️

Challenge 3: Tools and MCP

Time: ~30 minutes

Objectives: Add actions the agent can perform, such as creating mock IT tickets, and introduce MCP-based tool patterns.

Context: Some employee problems require action, not just answers.

Portal or SDK? Use the SDK to register functions and explore how tool calling works in Foundry.

Agents and Tools: Function tools, mock APIs, and MCP-oriented orchestration concepts.

Get Started: Create a mock ticket tool, define schemas, connect it to the agent, and test action-oriented requests.

Success Criteria: The agent correctly decides when to call a tool and returns useful ticket creation responses.

🧠

Challenge 4: Memory

Time: ~30 minutes

Objectives: Teach the agent to remember useful employee context such as department, preferences, and recent requests.

Context: Repeated users expect continuity and personalization.

Portal or SDK? Implement memory behavior in code and validate user experience in the portal.

Get Started: Define what to store, when to retrieve it, and how memory changes future answers.

Success Criteria: The agent uses remembered preferences appropriately without losing accuracy or safety.

🛡️

Challenge 5: Guardrails

Time: ~30 minutes

Objectives: Add controls that prevent unsafe responses, policy leakage, and prompt injection issues.

Context: Employee support agents operate in environments with sensitive data and policy boundaries.

Portal or SDK? Configure safety and policy handling in the agent logic and test failure modes.

Get Started: Create malicious or boundary-pushing prompts, refine instructions, and add response rules.

Success Criteria: The agent blocks unsafe requests and stays aligned with company policy requirements.

📏

Challenge 6: Evaluations

Time: ~30 minutes

Objectives: Measure correctness, groundedness, and helpfulness using repeatable evaluation scenarios.

Context: Enterprise teams need evidence before promoting an agent to wider use.

Portal or SDK? Use Foundry evaluation workflows and/or scripted evaluation datasets.

Get Started: Build a small evaluation set using synthetic employee support questions and compare results across versions.

Success Criteria: Evaluation metrics are collected and inform what should be improved next.

📈

Challenge 7: Optimization and Observability

Time: ~30 minutes

Objectives: Improve prompts, tools, and retrieval quality while tracing how the agent behaves.

Context: Better enterprise agents come from iterative improvement backed by trace data.

Portal or SDK? Tune in code, then inspect traces, logs, and tool behavior through observability workflows.

Get Started: Review weak responses, refine prompts or tools, rerun evaluations, and inspect trace outputs.

Success Criteria: Response quality improves and trace data clearly explains the agent’s reasoning path.

🚀

Challenge 8: Publish

Time: ~30 minutes

Objectives: Prepare the agent for enterprise rollout with the right deployment, governance, and support model.

Context: Publishing is not only deployment; it includes readiness, ownership, and operational trust.

Portal or SDK? Use both: finalize through code and validate publish-ready configuration in the portal.

Get Started: Review readiness criteria, confirm evaluations and guardrails, and define the enterprise publishing approach.

Success Criteria: The agent is documented, tested, governed, and ready for a controlled rollout.
