Use Case

Employee Support Agent

A practical enterprise scenario that maps cleanly to every part of the Foundry lifecycle.

Story

Employees ask questions every day such as:

  • How do I request vacation?
  • How do I reset my password?
  • Where is the travel policy?
  • Who approves expenses?

The AI agent should

  • Answer HR and IT questions
  • Search trusted internal documents
  • Create IT tickets through a mock API
  • Remember employee context and preferences
  • Follow company policy and safety boundaries
💼

Why this scenario fits every challenge

Challenge Scenario fit
1 · ModelsCompare stronger and smaller models for support use cases
2 · AgentCreate the baseline employee assistant
3 · GroundingAttach HR and IT knowledge base documents
4 · ToolsEnable ticket creation and action-taking flows
5 · MemoryRemember department, role, and preferences
6 · GuardrailsBlock policy leakage and unsafe requests
7 · EvaluationMeasure correctness, relevance, and groundedness
8 · OptimizationImprove prompts, tool descriptions, and retrieval
9 · ObservabilityTrace requests, latency, and failures
10 · PublishPrepare for governed enterprise rollout

This scenario also works well with the provided synthetic documents, mock API, and evaluation assets in starter-code/.

Challenge Hub

Everything on One Landing Page

Each challenge follows the same structure and links directly to the matching repo materials.

🧱

Challenge 0: Environment Setup

Time: ~30 minutes

Scenario: Before building the agent, your team needs a repeatable Azure environment.

Goal: Deploy the Azure Foundry environment using Bicep.

Instructions: Review infra/main.bicep, fill in deployment parameters, run the Azure deployment, and confirm resources in the portal.

Success Criteria: All required workshop resources are provisioned and visible.

Repo Links: Challenge README · Infra guide

🧠

Challenge 1: Model Choice Matters

Time: ~30 minutes

Scenario: The employee support assistant needs the right balance of quality, speed, and cost.

Goal: Compare two candidate models and choose the best fit.

Instructions: Test the same employee-support questions across models and record output quality, latency, and cost.

Success Criteria: A clear model decision is documented with evidence.

Repo Links: Challenge README

🤖

Challenge 2: Build Your First Agent

Time: ~45 minutes

Scenario: Start with a baseline Employee Support Agent driven by clear system instructions.

Goal: Create the first working agent in Microsoft Foundry.

Instructions: Create an agent, define its role and tone, set boundaries, and test it on employee questions.

Success Criteria: The baseline agent responds coherently to common HR and IT questions.

Repo Links: Challenge README

📚

Challenge 3: Ground the Agent

Time: ~45 minutes

Scenario: Employees need answers from trusted internal policies, not model guesswork.

Goal: Connect the agent to HR and IT knowledge sources.

Instructions: Upload the provided documents, attach them as a knowledge source, and compare grounded vs. ungrounded responses.

Success Criteria: Answers cite the provided documents and reduce hallucinations.

Repo Links: Challenge README · Starter code

🛠️

Challenge 4: Give the Agent Tools

Time: ~60 minutes

Scenario: Some requests require action, such as opening an IT ticket.

Goal: Register the mock ticketing API as a tool the agent can call.

Instructions: Review the mock API, register it as a tool, and test the full request-to-ticket flow.

Success Criteria: The agent calls the tool at the right time and returns a useful result.

Repo Links: Challenge README · Starter code

🧠

Challenge 5: Add Memory

Time: ~45 minutes

Scenario: The assistant should remember who the employee is and avoid asking the same questions again.

Goal: Add session and user memory for personalization.

Instructions: Decide what to store, how to retrieve it, and how remembered context changes later answers.

Success Criteria: The agent uses remembered context appropriately across conversations.

Repo Links: Challenge README

🛡️

Challenge 6: Make It Safe

Time: ~45 minutes

Scenario: The agent must refuse unsafe or confidential requests gracefully.

Goal: Add content safety, guardrails, and prompt injection protection.

Instructions: Configure safety controls, define confidential boundaries, and test prompt injection scenarios.

Success Criteria: Unsafe or disallowed requests are blocked with helpful refusal behavior.

Repo Links: Challenge README

📏

Challenge 7: Evaluate the Agent

Time: ~45 minutes

Scenario: Before broader rollout, you need evidence that the agent is reliable.

Goal: Measure groundedness, relevance, coherence, and correctness.

Instructions: Use the provided dataset or create your own, run evaluations, and identify failure cases.

Success Criteria: Evaluation results highlight strengths, weaknesses, and next actions.

Repo Links: Challenge README · Starter code

📈

Challenge 8: Optimize the Agent

Time: ~30 minutes

Scenario: Evaluation exposed weak spots in prompts, retrieval, or tool behavior.

Goal: Make targeted improvements and verify that the scores improve.

Instructions: Update prompts, tool descriptions, or chunking strategy, then rerun the same evaluation set.

Success Criteria: Quality improves measurably and the changes are documented.

Repo Links: Challenge README

📡

Challenge 9: Observe Everything

Time: ~30 minutes

Scenario: Once the agent is operating, the team needs traces, dashboards, and alerts to support it.

Goal: Configure tracing, monitoring, dashboards, and alerts.

Instructions: Enable Azure Monitor tracing, connect Application Insights, and verify request telemetry end-to-end.

Success Criteria: The team can diagnose behavior using telemetry and alerts.

Repo Links: Challenge README

🚀

Challenge 10: Enterprise Ready

Time: ~30 minutes

Scenario: The agent works, but it still needs governance and operational readiness for enterprise rollout.

Goal: Publish the agent with access controls, monitoring, and runbook readiness.

Instructions: Review publishing steps, confirm controls, and document operational ownership and rollback readiness.

Success Criteria: The agent is publish-ready for a controlled enterprise launch.

Repo Links: Challenge README
